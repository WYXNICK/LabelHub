# LabelHub 数据标注平台需求分析文档

## 1. 文档目的

本文档基于课题要求 PDF 与 `demo_data` 示例数据，沉淀 LabelHub 后续开发前的需求基线。当前阶段只定义业务目标、角色、核心功能、验收边界和关键约束，不进行编码实现和 Git 操作。

LabelHub 的产品目标是构建一套覆盖「数据生产 -> AI 预审 -> 人工审核 -> 多格式导出」全生命周期的 Web 数据标注平台。

## 2. 项目背景

大模型训练、评测和 Agent 调优依赖高质量业务数据。传统标注流程常见问题包括：

- 标注页面固定，无法快速适配不同类型数据集。
- 标注、审核、返修、导出链路割裂。
- 标注质量主要依赖人工抽查，缺少 AI 预审辅助。
- 数据导出格式与下游训练流程衔接成本高。
- 标注结果、AI 审核、人工审核之间缺少完整追溯。

LabelHub 需要通过动态模板、任务流转、AI 预审和多角色审核机制解决以上问题。

## 3. 总体目标

平台应支持以下完整链路：

1. 任务负责人创建任务，导入待标注数据。
2. 任务负责人通过模板搭建器配置标注页面。
3. 任务负责人配置 AI 审核标准并发布任务。
4. 标注员领取任务，在线作答，保存草稿并提交。
5. AI 审核 Agent 对提交结果进行自动预审。
6. 人工审核员进行复审或终审。
7. 打回结果返回标注员修改，通过结果进入可导出数据集。
8. 任务负责人按 JSON、JSONL、CSV、Excel 等格式导出数据。

## 4. 用户角色

| 角色 | 说明 | 核心能力 |
| --- | --- | --- |
| Owner 任务负责人 | 面向 PM、运营、数据生产负责人 | 创建任务、导入数据、搭建模板、配置审核、发布任务、查看结果、导出数据 |
| Labeler 标注员 | 面向标注员、行业专家 | 浏览任务、领取题目、在线作答、保存草稿、提交、查看打回意见并修改 |
| Reviewer 人工审核员 | 面向质检、复审、终审人员 | 查看待审数据、参考 AI 预审、通过、打回、修订、批量审核 |
| System / AI Agent | 后台系统角色 | 拉取已提交数据，调用 LLM 按规则预审，写回评分、结论和审计记录 |

最小实现必须包含 Owner、Labeler、Reviewer 三种人类角色。AI Agent 可以由后端 worker 承载，但需要具备独立系统账户视角，所有审核记录必须可追溯。

## 5. 核心业务需求

### 5.1 任务管理

Owner 可以创建和管理标注任务。

功能要求：

- 维护任务标题、描述、富文本说明、标签。
- 配置奖励规则、截止时间、题目配额。
- 支持任务状态：草稿、发布中、已暂停、已结束。
- 支持任务发布、暂停、恢复、结束。
- 支持发布前检查，至少校验任务基础信息、数据集、模板、审核配置是否完整。
- MVP 分发策略采用「先到先得」，后续可扩展指派、配额抢单。

验收标准：

- Owner 可以从空白任务完成创建、导入、配置模板、发布。
- 发布后 Labeler 能在任务广场看到任务。
- 暂停或结束后不能继续领取新题。

### 5.2 数据集管理

平台需要支持多格式数据导入。

功能要求：

- 支持 JSON、JSONL、Excel 三种导入格式。
- 导入后可以预览题目。
- 记录导入来源、格式、行号、错误信息。
- 支持导入校验，错误需定位到具体行和字段。
- 支持对重复数据进行识别，建议通过外部题目 ID 和 checksum 去重。

验收标准：

- `qa_quality` 数据集的 JSON、JSONL、Excel 均可导入并得到一致记录数。
- `preference_compare` 数据集的 JSON、JSONL、Excel 均可导入并得到一致记录数。
- 导入失败时不影响已成功导入的数据，并能展示错误详情。

### 5.3 动态标注模板

动态模板是本项目的核心难点之一。系统需要实现「模板搭建器 Designer」和「模板渲染器 Renderer」。

功能要求：

- Designer 包含左侧物料区、中间画布、右侧属性面板。
- 物料 schema 与实际渲染组件解耦。
- 模板搭建产物保存为可序列化 JSON Schema。
- 同一份 schema 可以在 Designer 预览，也可以在 Labeler 工作台运行。
- 支持模板版本管理，任务发布后绑定不可变模板版本。

至少支持的物料：

- 展示项 ShowItem
- 单行输入
- 多行文本
- 单选
- 多选
- 标签选择
- 富文本编辑器
- 文件上传
- 图片上传
- JSON 编辑器
- LLM 交互组件

进阶能力：

- 条件显示。
- 联动校验。
- 必填、长度、正则、JSON Schema 等校验规则。
- 分组容器。
- 多 Tab 布局。

验收标准：

- 能为问答质量标注搭建评分表单。
- 能为偏好对比标注搭建 A/B 对比表单。
- 模板发布后历史提交可以按当时版本复现。

### 5.4 标注员工作台

Labeler 需要通过工作台完成领取、作答、提交和返修。

功能要求：

- 任务广场支持搜索、筛选、任务卡片。
- 作答页面基于模板 schema 渲染。
- 支持上一题、下一题、跳题。
- 支持草稿自动保存。
- 支持提交校验与错误提示。
- 支持题目级 LLM 辅助调用。
- 支持查看已提交、通过、打回、待修改统计与列表。

验收标准：

- 刷新页面后草稿不丢失。
- 提交时必填、枚举、JSON 等校验能生效。
- 打回后 Labeler 能看到审核意见并再次修改提交。

### 5.5 AI 自动预审

AI 预审 Agent 是本项目第二个核心难点。

功能要求：

- Owner 可以配置审核 Prompt 模板。
- Owner 可以配置评分维度，例如相关性、准确性、格式合规、安全性。
- Labeler 提交后自动进入 AI 预审队列。
- Agent 调用大模型输出结构化结果。
- 结构化结果包含维度评分、结论、理由、建议。
- 结论至少包含通过、打回、人工复核。
- 失败需支持重试，超过重试次数进入人工复核。
- AI 审核结果与原始提交、Prompt 模板版本一并入库。

验收标准：

- Reviewer 可以看到 AI 评分与评语。
- AI 审核记录可以追溯到提交版本、模板版本和审核配置版本。
- AI 调用失败不会阻断人工审核流程。

### 5.6 人工审核流转

人工审核用于保证终审数据质量。

推荐流程：

```text
Labeler 提交 -> AI 预审
  -> AI 建议通过 -> 人工复审 -> 通过 -> 可导出
  -> AI 建议打回 -> Labeler 修改 或 人工复核
  -> AI 不确定/失败 -> 人工复核
```

功能要求：

- Reviewer 可查看待审列表。
- Reviewer 可查看原题、提交结果、AI 评语、历史审核意见。
- 支持第 1/2 轮提交 diff。
- 支持通过、打回。
- 打回必须填写理由。
- 支持批量审核。
- 所有状态迁移必须写入审计日志。

验收标准：

- 打回后 Labeler 能看到上一轮意见。
- 审核全过程可追溯。
- 终审通过的数据进入可导出集合。

### 5.7 多格式导出

Owner 需要将审核通过的数据交付给下游训练或评测流程。

功能要求：

- 至少支持 JSON、JSONL、CSV、Excel 四种格式。
- 支持异步导出。
- 支持导出历史列表。
- 支持导出进度查看。
- 支持字段映射：选择字段、字段重命名、是否包含审核记录。

验收标准：

- 四种格式均能导出已通过数据。
- 导出文件结构稳定，可被下游程序消费。
- 导出记录包含创建人、创建时间、导出参数、文件状态。

## 6. 非功能需求

### 6.1 工程结构

项目应采用前后端分离架构，并使用 monorepo 管理。

建议拆分：

- 前端应用：Owner 后台、Labeler 工作台、Reviewer 审核台。
- 后端 API：鉴权、任务、模板、数据集、审核、导出。
- Agent 服务：AI 预审 worker。
- 共享包：领域类型、模板 schema、UI 组件。

### 6.2 类型安全

- 推荐 TypeScript 全栈。
- 前后端共享核心枚举、DTO 和 schema。
- 禁止在关键领域逻辑中大量使用 `any`。

### 6.3 一致性与审计

- 所有状态迁移必须通过后端领域服务完成。
- 状态迁移必须在数据库事务中完成。
- 所有关键操作写入审计日志。
- AI 与人工审核结果都需要保留版本。

### 6.4 安全

- 不提交任何密钥、token、`.env`。
- PDF 中出现的示例 API key 不应进入代码或文档。
- 文件上传需要限制类型和大小。
- Markdown 渲染需要做白名单过滤。
- LLM 输出视为不可信输入，必须进行结构化校验。

### 6.5 产品体验

- 关键页面在 1280x800 与 1920x1080 下可用。
- 错误提示明确，能定位到字段或操作。
- 标注草稿自动保存，避免数据丢失。
- 审核和导出等长任务需展示进度。
