# LabelHub 开发粒度与实施计划

## 1. 文档目的

本文档基于 `docs/LabelHub_数据标注平台_AI全栈课题实现要求.md`、现有需求文档、Demo 数据分析、系统架构文档和前后端 SDD，说明 LabelHub 应从哪些部分开始开发，以及如何用较短流程完整覆盖官方功能要求。

核心原则：

- 先对齐 SDD 与接口契约，再开发。
- 先打通端到端闭环，再扩展物料、导出格式和体验细节。
- 每个阶段都必须同时明确前端页面、后端接口、状态迁移和验收标准。
- 开发粒度控制为 6 个阶段，避免把流程拆得过长。

## 2. 官方需求覆盖范围

必须完整覆盖的官方需求如下：

| 官方章节 | 功能点 | 本计划覆盖阶段 |
| --- | --- | --- |
| 角色与权限 | Owner、Labeler、Reviewer、System/AI Agent | 阶段 1、4 |
| 4.1 任务管理 | 基础信息、状态机、数据导入、批量编辑、题目预览、分发策略 | 阶段 1、2 |
| 4.2 动态模板 | Designer、Renderer、JSON Schema、全部物料、联动、校验、分组、Tab | 阶段 2 |
| 4.3 标注员工作台 | 任务广场、领取、作答、跳题、草稿、提交、LLM 辅助、我的数据 | 阶段 3 |
| 4.4 AI 自动预审 | Prompt 配置、评分维度、异步队列、结构化输出、重试、人工兜底 | 阶段 4 |
| 4.5 人工审核流转 | 复审/终审、diff、AI 评语、批量操作、打回理由、审计日志 | 阶段 4 |
| 4.6 多格式导出 | JSON、JSONL、CSV、Excel、异步导出、下载历史、字段映射 | 阶段 5 |
| 验收标准 | 全流程、工程质量、产品体验、文档与部署 | 阶段 0-5 |

## 3. 当前应优先从哪里开始

当前不应直接从页面堆砌开始，而应按下面顺序启动：

1. **先做阶段 0：契约与工程底座。**  
   原因：动态模板、提交、AI 审核、导出都依赖稳定的任务、模板、提交和审核状态。先把通用 SDD、OpenAPI、数据库迁移框架和统一错误定下来；业务状态机和实体枚举在进入对应阶段前通过 SDD 定稿，后面前后端才可以并行。

2. **再做阶段 1：Owner 任务、数据与发布前置校验。**  
   原因：所有后续功能都依赖任务、数据集、题目、模板版本、审核配置。没有任务和数据，Labeler、Agent、Reviewer 都没有可操作对象；但真实可领取发布必须等阶段 2 产出模板版本后才能放开。

3. **第三步做阶段 2：动态模板 Designer/Renderer。**  
   原因：这是官方最核心难点之一，且直接决定 Labeler 作答页和提交值结构。应在 Labeler 工作台之前完成最小可用模板运行时。

4. **然后做阶段 3 和阶段 4：标注提交 + AI/人工审核闭环。**  
   原因：这是验收中的主链路，也是展示项目含金量的关键。

5. **最后做阶段 5：导出、统计、体验和交付材料。**  
   原因：导出和统计依赖前面的终审通过数据，适合作为闭环之后的完整化阶段。

## 4. 开发阶段总览

| 阶段 | 目标 | 产出 |
| --- | --- | --- |
| 阶段 0 | SDD、接口、数据库和工程底座 | 可并行开发的契约和基础运行环境 |
| 阶段 1 | Owner 建任务、导入数据、审核配置、发布前检查 | Owner 可完成任务与数据准备；缺少模板版本时发布被明确阻塞 |
| 阶段 2 | 动态模板 Designer + Renderer | 模板可搭建、版本化、运行时渲染 |
| 阶段 3 | Labeler 任务广场、作答、草稿、提交 | Labeler 可完成提交和返修入口 |
| 阶段 4 | AI Agent 预审 + Reviewer 审核流转 | AI 预审、人工复审、打回、通过完整闭环 |
| 阶段 5 | 导出、看板、体验、部署和答辩材料 | 完整交付与演示能力 |

## 5. 阶段 0：SDD 契约与工程底座

### 5.1 目标

建立开发前的共同语言和技术底座，避免后续前端 VO 与后端返回字段不一致。

阶段 0 必须固化以下技术选型：React 18 + TypeScript、Ant Design、Formily + Schema 渲染、`@dnd-kit/core`、Zustand、Python、MySQL、OpenAI API 格式。

### 5.2 前端页面

此阶段不开发业务页面，只建立：

| 前端内容 | 说明 |
| --- | --- |
| 路由骨架 | `/owner/*`、`/labeler/*`、`/reviewer/*` |
| API Client 基础层 | 统一 base URL、错误处理、分页结构 |
| VO 类型目录 | `src/features/<domain>/types.ts` |
| 布局框架 | 登录后角色布局、侧边栏、顶部用户信息 |

### 5.3 后端功能与接口

| 后端功能 | 接口 |
| --- | --- |
| 健康检查 | `GET /api/health` |
| 当前用户 | `GET /api/auth/me` |
| 登录/登出 | `POST /api/auth/login`、`POST /api/auth/logout` |
| OpenAPI 文档 | `GET /api/openapi.json` |
| 通用错误结构 | 所有接口统一 `{ error: { code, message, details, requestId } }` |

### 5.4 数据与状态

- 初始化 `users`，包含 Owner、Labeler、Reviewer、System 四类账号；System 可作为服务账号视角，默认不开放人工登录。
- 阶段 0 只固化 `UserRole`、`UserStatus` 等认证与角色枚举；任务状态、提交状态、审核结论、导出状态等官方业务枚举必须在进入对应业务阶段前通过前后端 SDD 与数据库设计一起定稿。
- 建立 Alembic 迁移框架和 `users` 迁移；运行时认证可暂用内存 Demo 账号，MySQL 从阶段 1 开始成为任务、数据集、配置等业务主链路依赖。

### 5.5 验收标准

- 前后端 SDD 中的通用错误、分页、认证枚举、时间和 ID 类型已对齐。
- Python 后端能暴露 OpenAPI 或等价接口契约文档。
- 前端能根据 `GET /api/auth/me` 切换角色入口。
- 后端、Agent 均使用 `uv` 运行。

### 5.6 当前代码审查结论

结合当前仓库已实现代码，阶段 0 的工程底座已经基本完成：前端具备登录页、角色入口、受保护路由、统一 API Client 与登录后布局；后端具备健康检查、登录/登出/当前用户、HttpOnly Cookie Session、统一错误结构、请求追踪、OpenAPI 输出、`users` 迁移框架；Agent 具备 `uv` 工程、OpenAI API 格式配置入口和 AI 预审结果 DTO 雏形。

阶段 0 仍不是官方最终功能验收完成，只代表后续开发底座就绪。当前代码尚未实现官方 4.1-4.6 的任务、模板、标注、AI 预审、人工审核和导出业务闭环；这些必须在阶段 1-5 中按官方要求逐项覆盖。

## 6. 阶段 1：Owner 任务、数据集与发布前置校验

### 6.1 目标

让 Owner 能完成「建任务 -> 导入数据 -> 预览题目 -> 配置审核标准 -> 发布前检查」。阶段 1 不直接承诺可领取发布，因为官方要求任务发布链路必须绑定可运行的模板版本；模板 Designer/Renderer 在阶段 2 完成前，发布检查应明确返回“缺少模板版本”等阻塞项。

阶段 1 结束时应交付一个可被后续阶段复用的任务与数据底座：任务基础信息、状态机、数据集导入、题目预览、批量编辑、审核配置草稿/版本、发布检查和审计日志全部进入 MySQL，并进入 OpenAPI 契约。

### 6.2 SDD 与契约前置

阶段 1 开发前必须先更新 `apps/web/SDD.md` 与 `apps/api/SDD.md`，且字段名一一对齐。至少补齐以下契约：

| 契约类别 | 必须定义 |
| --- | --- |
| Task | `TaskStatus`、`DistributionStrategy`、`TaskVO`、`TaskDetailVO`、`CreateTaskRequest`、`UpdateTaskRequest`、`TaskStateTransitionRequest` |
| Dataset | `DatasetVO`、`DatasetItemVO`、`ImportJobVO`、`ImportStatus`、`ImportErrorRowVO`、`BatchUpdateDatasetItemsRequest` |
| ReviewConfig | `ReviewConfigDraftVO`、`ReviewConfigVersionVO`、`ReviewDimensionDTO`、`ReviewThresholdDTO` |
| PublishCheck | `PublishCheckVO`、`PublishBlockerVO`、阻塞项编码，如 `MISSING_DATASET`、`MISSING_TEMPLATE_VERSION`、`MISSING_REVIEW_CONFIG` |
| Audit | `AuditLogVO`、状态迁移与批量编辑审计字段 |

### 6.3 前端页面

| 页面 | 路由 | 功能 |
| --- | --- | --- |
| 任务管理列表 | `/owner/tasks` | 搜索、筛选、状态标签、创建入口、暂停/结束入口、最近更新时间和数据量摘要 |
| 任务创建/编辑 | `/owner/tasks/new`、`/owner/tasks/:taskId/settings` | 标题、描述、富文本说明、标签、奖励规则、截止时间、配额、分发策略 |
| 数据集管理 | `/owner/tasks/:taskId/datasets` | 上传 JSON/JSONL/Excel、导入进度、导入错误行、题目预览、批量启用/禁用或标签编辑 |
| 审核配置 | `/owner/tasks/:taskId/review-config` | Prompt 模板、评分维度、通过/打回/人工复核阈值；只做配置版本，不触发 Agent 执行 |
| 发布检查抽屉 | 任意 Owner 任务页内弹出 | 检查基础信息、数据集、模板版本、审核配置、配额、截止时间；阶段 1 必须能展示缺少模板版本并阻止发布 |
| 任务审计时间线 | `/owner/tasks/:taskId/audit` 或任务详情内嵌 | 展示创建、编辑、导入、批量编辑、状态迁移、审核配置发布等关键操作 |

| 功能 | 接口 |
| --- | --- |
| 任务列表 | `GET /api/tasks?page=&pageSize=&status=&keyword=` |
| 创建任务 | `POST /api/tasks` |
| 任务详情 | `GET /api/tasks/{taskId}` |
| 更新任务 | `PATCH /api/tasks/{taskId}` |
| 状态迁移 | `POST /api/tasks/{taskId}/state-transitions` |
| 发布前检查 | `GET /api/tasks/{taskId}/publish-check` |
| 创建文件上传记录 | `POST /api/files` |
| 创建导入任务 | `POST /api/tasks/{taskId}/import-jobs` |
| 查询导入任务 | `GET /api/import-jobs/{importJobId}` |
| 查询导入错误 | `GET /api/import-jobs/{importJobId}/errors?page=&pageSize=` |
| 数据集列表 | `GET /api/tasks/{taskId}/datasets` |

### 6.5 数据库与状态机要求

阶段 1 必须把任务与数据主链路落到 MySQL。建议最小实体包括：

| 实体 | 说明 |
| --- | --- |
| `tasks` | 任务基础信息、状态、分发策略、配额、截止时间、Owner |
| `task_state_transitions` / `audit_logs` | 状态迁移和关键操作审计 |
| `files` | 导入文件元信息、校验摘要、文件类型、大小 |
| `datasets` | 任务下的数据集版本与统计 |
| `dataset_items` | 原始题目 payload、启用状态、标签、顺序号 |
| `import_jobs` | 导入任务状态、进度、成功/失败计数 |
| `import_error_rows` | 导入错误行、错误原因、原始片段 |
| `review_config_drafts` | 审核配置草稿 |
| `review_config_versions` | 已发布审核配置版本，供阶段 4 Agent 读取 |

任务状态至少包含官方要求的 `DRAFT`、`PUBLISHED`、`PAUSED`、`ENDED`。阶段 1 中 `PUBLISHED` 状态迁移必须受发布检查保护；缺少模板版本时返回 409 或等价业务错误，不得产生可领取任务。

### 6.6 开发粒度

| 粒度 | 内容 | 验收 |
| --- | --- | --- |
| 1.0 | 阶段 1 SDD、OpenAPI、Entity 与 Alembic 迁移 | 前后端字段映射一致；任务、数据集、导入、审核配置、审计实体进入迁移脚本 |
| 1.1 | 任务 CRUD + 状态机 | 草稿、发布中、暂停、结束规则清晰；非法迁移返回 409；所有迁移写审计 |
| 1.2 | JSON/JSONL/Excel 导入 | `qa_quality` 30 条、`preference_compare` 12 条可导入；错误行可追踪；重复导入具备幂等或去重策略 |
| 1.3 | 数据预览与批量编辑 | 可查看 payload，批量启用/禁用或修改标签；批量操作写审计 |
| 1.4 | 审核配置草稿与版本 | Prompt、维度、阈值可保存和发布版本；只产生配置，不触发 AI 预审 |
| 1.5 | 发布前检查 | 缺少数据集、模板版本、审核配置、配额或截止时间时不能发布；阶段 1 必须明确阻塞缺少模板版本 |

### 6.7 覆盖需求

- 任务基础信息。
- 任务状态机。
- 题目导入 JSON/JSONL/Excel。
- 批量编辑、题目预览。
- 分发策略选用官方允许的「先到先得」作为 MVP，并在任务实体中预留分配/配额策略扩展点。
- 审核 Prompt 与评分维度配置，作为阶段 4 AI Agent 的输入配置源。
- 发布前抽屉必须按官方任务发布要求展示配额、截止时间、数据集、模板版本和审核配置的完整性。

### 6.8 阶段 1 边界

阶段 1 不开发动态模板 Designer/Renderer，不开发 Labeler 领取与作答，不开发 AI Agent 异步预审，不开发 Reviewer 人工复核和多格式导出。阶段 1 只为这些后续链路提供稳定的任务、数据集、审核配置和审计底座。

如果阶段 1 需要演示“发布检查”，应演示一个阻塞式结果：任务基础信息、数据集和审核配置已完成，但由于阶段 2 尚未发布模板版本，系统清晰提示不能进入可领取发布状态。这比临时绕过模板更符合官方最终要求。

### 6.9 当前代码审查结论

结合当前仓库实现，阶段 1 的 1.0-1.5 已完整落地：任务 CRUD、状态机、数据集导入、题目预览、批量编辑、审核配置草稿/版本、发布前检查和审计日志均已进入前后端代码与 OpenAPI 契约，并在阶段 1 起使用 MySQL 作为主链路数据库。

阶段 1 的发布检查保持官方要求边界：即使任务基础信息、数据集和审核配置都已就绪，只要阶段 2 尚未发布模板版本，系统仍必须返回 `MISSING_TEMPLATE_VERSION` 并阻止任务进入可领取发布状态。当前实现没有绕过模板要求，因此阶段 1 可以认为是“Owner 前置准备闭环完成”，不是官方最终全链路完成。

## 7. 阶段 2：动态模板 Designer 与 Renderer

### 7.1 目标

实现官方最核心难点：同一份模板 schema 既能被 Owner 拖拽搭建，也能被 Labeler 运行时渲染。

阶段 2 开始前必须先更新 `apps/web/SDD.md` 与 `apps/api/SDD.md`，定稿模板 Schema、版本规则、物料字段和 Renderer 输入输出。阶段 2 完成后，任务发布检查中的 `MISSING_TEMPLATE_VERSION` 才能被真实模板版本解除。

### 7.2 前端页面

| 页面 | 路由 | 功能 |
| --- | --- | --- |
| 模板搭建器 | `/owner/tasks/:taskId/designer` | 左侧物料、中间画布、右侧属性面板、保存草稿、校验和发布入口 |
| 模板预览 | Designer 内嵌抽屉或右侧 Tab | 选择当前任务 demo item，用同一 Renderer 运行模板 schema |
| 模板版本列表 | Designer 侧栏或 `/owner/tasks/:taskId/template-versions` | 版本号、发布时间、发布人、启用状态、绑定到任务当前模板版本 |

### 7.3 后端功能与接口

| 功能 | 接口 |
| --- | --- |
| 获取模板草稿 | `GET /api/tasks/{taskId}/template-draft` |
| 保存模板草稿 | `PUT /api/tasks/{taskId}/template-draft` |
| 校验模板 schema | `POST /api/template-schemas:validate` |
| 发布模板版本 | `POST /api/tasks/{taskId}/template-versions` |
| 模板版本列表 | `GET /api/tasks/{taskId}/template-versions` |
| 获取模板版本 | `GET /api/template-versions/{templateVersionId}` |

### 7.4 数据库与契约要求

阶段 2 至少新增以下实体或等价结构：

| 实体 | 说明 |
| --- | --- |
| `template_drafts` | 任务当前模板草稿，保存可编辑 schema、最近保存人和时间 |
| `template_versions` | 已发布不可变模板版本，发布后绑定 `tasks.current_template_version_id` |

模板 Schema 建议保持语言无关 JSON，至少包含：

- `schemaVersion`：内部模板协议版本。
- `components`：物料列表，每个物料包含 `id`、`type`、`fieldKey`、`label`、`props`、`validation`、`visibility`。
- `layout`：画布顺序、分组容器、Tab 容器结构。
- `llmActions`：LLM 交互组件的 prompt、输入字段映射、输出字段映射。
- `showItems`：展示题目原始数据的只读绑定路径。

阶段 2 的后端校验必须保证：

- `component.id` 与可提交字段 `fieldKey` 唯一。
- 物料 `type` 只能来自官方要求物料白名单。
- 发布版本不可变，后续修改只能生成新版本。
- 发布模板版本后更新 `tasks.current_template_version_id` 和 `tasks.version`，从而让阶段 1.5 发布检查可以解除模板阻塞。

### 7.5 开发粒度

| 粒度 | 内容 | 验收 |
| --- | --- | --- |
| 2.0 | 阶段 2 SDD、OpenAPI、Entity 与 Alembic 迁移 | 已完成：前后端模板字段映射一致；`template_drafts`、`template_versions` 已进入迁移脚本 |
| 2.1 | 模板 schema 基础结构和后端校验 | 已完成：schema 可保存、校验失败有结构化错误；非法物料、重复字段和非法布局会被拒绝 |

## 14. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 先写页面导致接口字段漂移 | 前后端返工 | 每阶段先更新两份 SDD，再编码 |
| 阶段 1 绕过模板提前发布可领取任务 | 后续 Labeler 无法按 schema 作答，偏离官方要求 | 发布检查必须阻塞缺少模板版本；真正可领取发布等阶段 2 模板版本完成后放开 |
| 阶段 1 把审核配置误做成 Agent 执行 | 阶段边界膨胀，影响主链路节奏 | 阶段 1 只做 Prompt、维度、阈值配置版本；异步队列和 LLM 调用放到阶段 4 |
| Designer 过大 | 延误主链路 | 阶段 2 先完成 schema + 基础物料，再补高级物料 |
| AI 输出不稳定 | 审核流卡住 | Pydantic 校验、重试、人工复核 |
| 状态流转遗漏 | 数据不可追溯 | 所有迁移走后端服务并写审计 |
| 导出字段混乱 | 下游不可消费 | 字段映射保存版本，导出快照固定 |
| Demo 数据覆盖不足 | 验收展示单薄 | 同时导入 `qa_quality` 和 `preference_compare`，覆盖文本、图片、视频、Markdown |

## 15. 阶段完成判定

一个阶段只有同时满足以下条件才算完成：

- 前端页面可操作。
- 后端接口有 Request/VO/DTO 定义并进入 OpenAPI。
- 前后端字段映射通过 SDD 检查。
- 关键状态迁移有审计日志。
- 至少有单元测试或集成测试覆盖核心服务。
- 至少有一条 Demo 数据完成该阶段主流程。
- 涉及前端页面或核心交互时，必须完成 Chrome DevTools MCP 真实浏览器验收。

### 15.1 前端真实浏览器验收要求

从阶段 1.1 起，每个涉及前端页面或核心交互的开发粒度，在代码检查通过后都必须补充真实浏览器验收。验收工具优先使用 Chrome DevTools MCP，并且后端必须连接 MySQL；数据库测试不得用 SQLite 代替 MySQL。

最低验收清单：

- 运行后端 API 时，`DATABASE_URL` 必须指向 MySQL，并确认 Alembic 已升级到当前 head。
- 运行前端时，确认 API base URL、CORS origin 与 Cookie host 一致；本地验收优先统一使用 `localhost`，不要在同一链路混用 `127.0.0.1`。
- 使用 Chrome DevTools MCP 检查 `1280×800` 与 `1920×1080` 两个视口。
- 每个视口至少检查页面首屏、主要表单/表格、主要按钮交互、loading/empty/error 状态、关键弹窗或抽屉。
- 检查 Console 与 Network；非预期 error/issue 必须修复。业务预期错误，例如发布保护 `409 PUBLISH_BLOCKED`，必须确认页面展示的阻塞项与后端返回一致。
- 对涉及写入的页面，必须通过 Network 或数据库侧确认数据进入 MySQL。

阶段 1.1-1.5 均已按该标准完成真实验收：任务管理、数据导入与批量编辑、审核配置、发布前检查均通过 Chrome DevTools MCP 检查，并在 MySQL 中确认关键数据落库。

## 16. 总结

当前阶段 0、阶段 1、阶段 2.0、阶段 2.1 与阶段 2.2 已完成，下一步最合理的开发起点是阶段 2.3 Designer 三栏布局与拖拽排序。它会把 2.2 已验证可运行的 Renderer 接入真实搭建流程；阶段 2 完成模板版本后，再依次完成 Labeler 提交、AI 自动预审、Reviewer 审核和导出。

这条路线阶段不多，但每个阶段都能交付可验证的业务能力；同时完整覆盖官方需求中的 Owner 后台、Labeler 工作台、AI Agent、人工审核流转和多格式导出。
