# LabelHub 开发粒度与实施计划

## 1. 文档目的

本文档基于 `docs/LabelHub_数据标注平台_AI全栈课题实现要求.md`、现有需求文档、Demo 数据分析、系统架构文档和前后端 SDD，说明 LabelHub 应从哪些部分开始开发，以及如何用较短流程完整覆盖官方功能要求。

核心原则：

- 先对齐 SDD 与接口契约，再开发。
- 先打通端到端闭环，再扩展物料、导出格式和体验细节。
- 每个阶段都必须同时明确前端页面、后端接口、状态迁移和验收标准。
- 开发粒度控制为 6 个阶段，避免把流程拆得过长。

## 2. 官方需求覆盖范围

必须完整覆盖的官方需求如下：

| 官方章节 | 功能点 | 本计划覆盖阶段 |
| --- | --- | --- |
| 角色与权限 | Owner、Labeler、Reviewer、System/AI Agent | 阶段 1、4 |
| 4.1 任务管理 | 基础信息、状态机、数据导入、批量编辑、题目预览、分发策略 | 阶段 1、2 |
| 4.2 动态模板 | Designer、Renderer、JSON Schema、全部物料、联动、校验、分组、Tab | 阶段 2 |
| 4.3 标注员工作台 | 任务广场、领取、作答、跳题、草稿、提交、LLM 辅助、我的数据 | 阶段 3 |
| 4.4 AI 自动预审 | Prompt 配置、评分维度、异步队列、结构化输出、重试、人工兜底 | 阶段 4 |
| 4.5 人工审核流转 | 复审/终审、diff、AI 评语、批量操作、打回理由、审计日志 | 阶段 4 |
| 4.6 多格式导出 | JSON、JSONL、CSV、Excel、异步导出、下载历史、字段映射 | 阶段 5 |
| 验收标准 | 全流程、工程质量、产品体验、文档与部署 | 阶段 0-5 |

## 3. 当前应优先从哪里开始

当前不应直接从页面堆砌开始，而应按下面顺序启动：

1. **先做阶段 0：契约与工程底座。**  
   原因：动态模板、提交、AI 审核、导出都依赖稳定的任务、模板、提交和审核状态。先把 SDD、OpenAPI、数据库迁移、统一错误和枚举定下来，后面前后端才可以并行。

2. **再做阶段 1：Owner 任务与数据闭环。**  
   原因：所有后续功能都依赖任务、数据集、题目、模板版本、审核配置。没有任务和数据，Labeler、Agent、Reviewer 都没有可操作对象。

3. **第三步做阶段 2：动态模板 Designer/Renderer。**  
   原因：这是官方最核心难点之一，且直接决定 Labeler 作答页和提交值结构。应在 Labeler 工作台之前完成最小可用模板运行时。

4. **然后做阶段 3 和阶段 4：标注提交 + AI/人工审核闭环。**  
   原因：这是验收中的主链路，也是展示项目含金量的关键。

5. **最后做阶段 5：导出、统计、体验和交付材料。**  
   原因：导出和统计依赖前面的终审通过数据，适合作为闭环之后的完整化阶段。

## 4. 开发阶段总览

| 阶段 | 目标 | 产出 |
| --- | --- | --- |
| 阶段 0 | SDD、接口、数据库和工程底座 | 可并行开发的契约和基础运行环境 |
| 阶段 1 | Owner 建任务、导入数据、审核配置、发布 | Owner 可发布一个可领取任务 |
| 阶段 2 | 动态模板 Designer + Renderer | 模板可搭建、版本化、运行时渲染 |
| 阶段 3 | Labeler 任务广场、作答、草稿、提交 | Labeler 可完成提交和返修入口 |
| 阶段 4 | AI Agent 预审 + Reviewer 审核流转 | AI 预审、人工复审、打回、通过完整闭环 |
| 阶段 5 | 导出、看板、体验、部署和答辩材料 | 完整交付与演示能力 |

## 5. 阶段 0：SDD 契约与工程底座

### 5.1 目标

建立开发前的共同语言和技术底座，避免后续前端 VO 与后端返回字段不一致。

阶段 0 必须固化以下技术选型：React 18 + TypeScript、Ant Design、Formily + Schema 渲染、`@dnd-kit/core`、Zustand、Python、MySQL、OpenAI API 格式。

### 5.2 前端页面

此阶段不开发业务页面，只建立：

| 前端内容 | 说明 |
| --- | --- |
| 路由骨架 | `/owner/*`、`/labeler/*`、`/reviewer/*` |
| API Client 基础层 | 统一 base URL、错误处理、分页结构 |
| VO 类型目录 | `src/features/<domain>/types.ts` |
| 布局框架 | 登录后角色布局、侧边栏、顶部用户信息 |

### 5.3 后端功能与接口

| 后端功能 | 接口 |
| --- | --- |
| 健康检查 | `GET /api/health` |
| 当前用户 | `GET /api/auth/me` |
| 登录/登出 | `POST /api/auth/login`、`POST /api/auth/logout` |
| OpenAPI 文档 | `GET /api/openapi.json` |
| 通用错误结构 | 所有接口统一 `{ error: { code, message, details, requestId } }` |

### 5.4 数据与状态

- 初始化 `users`，包含 Owner、Labeler、Reviewer、System 四类测试账号。
- 定义核心枚举：任务状态、提交状态、审核结论、导出状态。
- 建立 Alembic 迁移框架。

### 5.5 验收标准

- 前后端 SDD 中的通用错误、分页、枚举、时间和 ID 类型已对齐。
- Python 后端能暴露 OpenAPI 或等价接口契约文档。
- 前端能根据 `GET /api/auth/me` 切换角色入口。
- 后端、Agent 均使用 `uv` 运行。

## 6. 阶段 1：Owner 任务、数据集与发布闭环

### 6.1 目标

让 Owner 能完成「建任务 -> 导入数据 -> 预览题目 -> 配置审核 -> 发布任务」。

### 6.2 前端页面

| 页面 | 路由 | 功能 |
| --- | --- | --- |
| 任务管理列表 | `/owner/tasks` | 搜索、筛选、状态标签、创建入口、暂停/结束入口 |
| 任务创建/编辑 | `/owner/tasks/new`、`/owner/tasks/:taskId/settings` | 标题、描述、富文本说明、标签、奖励规则、截止时间、配额 |
| 数据集管理 | `/owner/tasks/:taskId/datasets` | 上传 JSON/JSONL/Excel、导入进度、错误行、题目预览、批量编辑入口 |
| 审核配置 | `/owner/tasks/:taskId/review-config` | Prompt 模板、评分维度、通过/打回/人工复核阈值 |
| 发布检查抽屉 | 任意 Owner 任务页内弹出 | 检查基础信息、数据集、模板、审核配置、配额、截止时间 |

### 6.3 后端功能与接口

| 功能 | 接口 |
| --- | --- |
| 任务列表 | `GET /api/tasks?page=&pageSize=&status=&keyword=` |
| 创建任务 | `POST /api/tasks` |
| 任务详情 | `GET /api/tasks/{taskId}` |
| 更新任务 | `PATCH /api/tasks/{taskId}` |
| 状态迁移 | `POST /api/tasks/{taskId}/state-transitions` |
| 创建文件上传记录 | `POST /api/files` |
| 创建导入任务 | `POST /api/tasks/{taskId}/import-jobs` |
| 查询导入任务 | `GET /api/import-jobs/{importJobId}` |
| 数据集列表 | `GET /api/tasks/{taskId}/datasets` |
| 题目预览 | `GET /api/datasets/{datasetId}/items?page=&pageSize=&keyword=` |
| 批量编辑题目 | `PATCH /api/datasets/{datasetId}/items:batch` |
| 保存审核配置 | `PUT /api/tasks/{taskId}/review-config-draft` |
| 发布审核配置 | `POST /api/tasks/{taskId}/review-config-versions` |

### 6.4 开发粒度

| 粒度 | 内容 | 验收 |
| --- | --- | --- |
| 1.1 | 任务 CRUD + 状态机 | 草稿、发布中、暂停、结束可迁移，非法迁移返回 409 |
| 1.2 | JSON/JSONL/Excel 导入 | `qa_quality` 30 条、`preference_compare` 12 条可导入 |
| 1.3 | 数据预览与批量编辑 | 可查看 payload，批量启用/禁用或修改标签 |
| 1.4 | 审核配置与发布检查 | 缺少数据集/模板/审核配置不能发布 |

### 6.5 覆盖需求

- 任务基础信息。
- 任务状态机。
- 题目导入 JSON/JSONL/Excel。
- 批量编辑、题目预览。
- 分发策略选用「先到先得」作为 MVP。
- 审核 Prompt 与评分维度配置。

## 7. 阶段 2：动态模板 Designer 与 Renderer

### 7.1 目标

实现官方最核心难点：同一份模板 schema 既能被 Owner 拖拽搭建，也能被 Labeler 运行时渲染。

### 7.2 前端页面

| 页面 | 路由 | 功能 |
| --- | --- | --- |
| 模板搭建器 | `/owner/tasks/:taskId/designer` | 左侧物料、中间画布、右侧属性面板 |
| 模板预览 | Designer 内嵌或 `/owner/tasks/:taskId/designer/preview` | 用 demo item 运行 Renderer |
| 模板版本列表 | `/owner/tasks/:taskId/template-versions` | 版本号、发布时间、发布人、启用状态 |

### 7.3 后端功能与接口

| 功能 | 接口 |
| --- | --- |
| 获取模板草稿 | `GET /api/tasks/{taskId}/template-draft` |
| 保存模板草稿 | `PUT /api/tasks/{taskId}/template-draft` |
| 校验模板 schema | `POST /api/template-schemas:validate` |
| 发布模板版本 | `POST /api/tasks/{taskId}/template-versions` |
| 模板版本列表 | `GET /api/tasks/{taskId}/template-versions` |
| 获取模板版本 | `GET /api/template-versions/{templateVersionId}` |

### 7.4 开发粒度

| 粒度 | 内容 | 验收 |
| --- | --- | --- |
| 2.1 | 模板 schema 基础结构和校验 | schema 可保存、校验、发布版本 |
| 2.2 | Designer 三栏布局与拖拽排序 | 左侧物料能拖到画布，画布能排序/删除 |
| 2.3 | 基础物料 | 单行、多行、单选、多选、标签选择、ShowItem |
| 2.4 | 高级物料 | 富文本、文件/图片上传、JSON 编辑器、LLM 交互组件 |
| 2.5 | 高级布局和规则 | 条件显示、联动校验、必填/长度/正则/自定义规则、分组、Tab |
| 2.6 | Renderer 运行时 | 同一 schema 在 Designer 预览和 Labeler 作答页一致渲染 |

### 7.5 覆盖需求

- Designer + Renderer。
- 可序列化 JSON Schema。
- 所有官方要求物料。
- 条件显示、联动校验、自定义校验。
- 分组容器、多 Tab。
- 模板发布后版本不可变。

## 8. 阶段 3：Labeler 工作台与提交闭环

### 8.1 目标

让 Labeler 能完成「看任务 -> 领取题目 -> 作答 -> 草稿保存 -> 提交 -> 查看打回」。

### 8.2 前端页面

| 标注工作台 | `/labeler/assignments/:assignmentId` | 题目导航、Renderer 表单、草稿状态、提交校验、LLM 辅助 |
| 我的贡献 | `/labeler/contributions` | 已提交、通过、打回、待修改统计和列表 |
| 打回修改详情 | `/labeler/assignments/:assignmentId/revise` | 审核意见、上一轮提交、修改再提交 |

### 8.3 后端功能与接口

| 功能 | 接口 |
| --- | --- |
| 任务广场 | `GET /api/marketplace/tasks?page=&pageSize=&keyword=&tag=` |
| 领取题目 | `POST /api/tasks/{taskId}/assignments` |
| 我的领取列表 | `GET /api/assignments?status=&page=&pageSize=` |
| 作答上下文 | `GET /api/assignments/{assignmentId}` |
| 保存草稿 | `PUT /api/assignments/{assignmentId}/draft` |
| 提交标注 | `POST /api/assignments/{assignmentId}/submissions` |
| 题目级 LLM 辅助 | `POST /api/llm-actions/{actionId}/runs` |
| 我的贡献统计 | `GET /api/me/contribution-stats` |

### 8.4 开发粒度

| 粒度 | 内容 | 验收 |
| --- | --- | --- |
| 3.1 | 任务广场与先到先得领取 | 发布中任务可见，领取受配额和截止时间约束 |
| 3.2 | 作答上下文和题目导航 | 上一题/下一题/跳题可用 |
| 3.3 | 草稿自动保存 | 刷新页面后恢复草稿 |
| 3.4 | 提交校验和提交版本 | 必填、枚举、JSON 校验生效，提交生成版本 |
| 3.5 | 我的贡献和打回修改入口 | 打回原因可见，可再次提交 |
| 3.6 | 题目级 LLM 辅助 | LLM 输出仅作参考或预填，Labeler 确认后才提交 |

### 8.5 覆盖需求

- 任务广场搜索/筛选/卡片。
- 领取任务。
- 上一题/下一题/跳题。
- 草稿自动保存。
- 提交校验。
- 题目级 LLM 辅助。
- 我的数据统计和列表。

## 9. 阶段 4：AI 自动预审与人工审核流转

### 9.1 目标

打通「提交 -> AI 预审 -> Reviewer 复审/终审 -> 打回或通过 -> 审计追溯」。

### 9.2 前端页面

| 页面 | 路由 | 功能 |
| --- | --- | --- |
| 审核工作台 | `/reviewer/reviews` | 待审列表、任务筛选、AI 结论筛选、批量操作 |
| 审核详情 | `/reviewer/reviews/:reviewId` | 原题、提交值、AI 评分、AI 评语、Prompt 摘要、diff、时间线 |
| 审核结果列表 | `/reviewer/review-results` | 已通过、已打回、人工复核历史 |
| Owner 数据验收 | `/owner/tasks/:taskId/acceptance` | 任务级通过率、打回率、AI 结论分布、待审数据 |

### 9.3 后端功能与接口

| 功能 | 接口 |
| --- | --- |
| 创建 AI job | 提交接口内部触发 |
| Agent 领取 job | `POST /api/internal/review-jobs/{jobId}/claims` |
| Agent 写回结果 | `POST /api/internal/review-jobs/{jobId}/results` |
| AI job 查询 | `GET /api/review-jobs?status=&page=&pageSize=` |
| 待审列表 | `GET /api/reviews?status=&taskId=&aiConclusion=&page=&pageSize=` |
| 审核详情 | `GET /api/reviews/{reviewId}` |
| 审核决策 | `POST /api/reviews/{reviewId}/decisions` |
| 批量审核 | `POST /api/reviews:batch-decide` |
| 审计时间线 | `GET /api/audit-logs?entityType=&entityId=` |
| Owner 验收统计 | `GET /api/tasks/{taskId}/acceptance-stats` |

### 9.4 开发粒度

| 粒度 | 内容 | 验收 |
| --- | --- | --- |
| 4.1 | Review job 与幂等入队 | 同一提交重复触发不会生成重复有效 AI 结论 |
| 4.2 | Agent OpenAI API 格式调用 | 结构化输出通过后端模型校验，失败可重试 |
| 4.3 | AI 结果写回和状态推进 | PASS、RETURN、NEEDS_HUMAN_REVIEW 映射到审核流 |
| 4.4 | Reviewer 待审与详情 | 能查看原题、提交、AI 评语、历史意见 |
| 4.5 | 通过、打回、批量审核 | 打回必须填写理由，批量操作写每条审计 |
| 4.6 | diff 和审计时间线 | 第 1/2 轮提交 diff、AI/人工操作可追溯 |

### 9.5 覆盖需求

- 可配置评测标准。
- 异步任务队列。
- Function Calling/结构化输出。
- 失败重试和人工兜底。
- AI 评语与原始 Prompt 可查。
- 人工复审/终审。
- 批量操作。
- 打回理由。
- 审计日志。

| LlmAction | 题目级 LLM 辅助 |
| ReviewConfig | Prompt 模板、评分维度、阈值、版本 |
| AgentJob | AI job 入队、领取、重试、写回 |
| Review | AI 结果、人工审核、批量操作、打回、通过 |
| Audit | 状态迁移、操作时间线 |
| Export | JSON/JSONL/CSV/Excel、字段映射、下载历史 |
| Dashboard | Owner、Labeler、Reviewer 统计 |
| File | 导入文件、证据文件、导出文件 |

## 13. 推荐并行策略

只有在阶段 0 的 SDD 和接口契约对齐后，才允许并行。

| 可并行开发 | 前置条件 |
| --- | --- |
| Owner 页面与 Task API | TaskVO、CreateTaskRequest、状态枚举已对齐 |
| Dataset 导入与前端预览 | ImportJobVO、DatasetItemVO 已对齐 |
| Designer UI 与 Template API | TemplateSchema JSON 结构已对齐 |
| Labeler 工作台与 Assignment/Submission API | AssignmentVO、SubmissionVO 已对齐 |
| Agent worker 与 Reviewer UI | AiReviewResultDTO、ReviewVO 已对齐 |
| Export UI 与 Export worker | ExportJobVO、字段映射结构已对齐 |

不允许并行的部分：

- 数据库迁移和 Entity 基础定义。
- 状态机枚举与状态迁移规则。
- Template Schema 协议。
- AI 输出 DTO 和审核结论映射。

## 14. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 先写页面导致接口字段漂移 | 前后端返工 | 每阶段先更新两份 SDD，再编码 |
| Designer 过大 | 延误主链路 | 阶段 2 先完成 schema + 基础物料，再补高级物料 |
| AI 输出不稳定 | 审核流卡住 | Pydantic 校验、重试、人工复核 |
| 状态流转遗漏 | 数据不可追溯 | 所有迁移走后端服务并写审计 |
| 导出字段混乱 | 下游不可消费 | 字段映射保存版本，导出快照固定 |
| Demo 数据覆盖不足 | 验收展示单薄 | 同时导入 `qa_quality` 和 `preference_compare`，覆盖文本、图片、视频、Markdown |

## 15. 阶段完成判定

一个阶段只有同时满足以下条件才算完成：

- 前端页面可操作。
- 后端接口有 Request/VO/DTO 定义并进入 OpenAPI。
- 前后端字段映射通过 SDD 检查。
- 关键状态迁移有审计日志。
- 至少有单元测试或集成测试覆盖核心服务。
- 至少有一条 Demo 数据完成该阶段主流程。

## 16. 总结

当前最合理的开发起点是阶段 0 和阶段 1：先把 SDD、契约、数据库、Auth、任务、数据集和发布检查做稳。随后进入动态模板，因为它是 Labeler 作答和提交结构的基础。再往后依次完成 Labeler 提交、AI 自动预审、Reviewer 审核和导出。

这条路线阶段不多，但每个阶段都能交付可验证的业务能力；同时完整覆盖官方需求中的 Owner 后台、Labeler 工作台、AI Agent、人工审核流转和多格式导出。
