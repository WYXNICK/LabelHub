# LabelHub 技术选型基线

## 1. 文档目的

本文档记录 LabelHub 当前采用的技术选型和执行约束。后续如需调整技术栈，应同步更新系统架构、前后端 SDD、部署说明和测试策略。

## 2. 已确定选型

| 层 | 技术选型 | 说明 |
| --- | --- | --- |
| 前端框架 | React 18 + TypeScript | 支撑多角色工作台、动态模板和类型安全 |
| UI 组件库 | Ant Design | 用于表单、表格、抽屉、弹窗、标签和反馈组件 |
| 动态模板运行时 | 自研 TemplateSchemaVO + React Renderer + Ant Design 控件 | 服务模板搭建、运行时渲染、提交值收集与校验 |
| 拖拽 | @dnd-kit/core | 服务模板搭建器物料拖拽和排序 |
| 状态管理 | Zustand | 管理登录态、角色入口和局部跨页面状态 |
| 后端框架 | Python + FastAPI | 提供 REST API、OpenAPI 和异步能力 |
| 数据模型 | Pydantic v2 | 请求、响应、LLM 输出和内部 DTO 校验 |
| ORM | SQLAlchemy 2 | 统一数据访问和事务边界 |
| 数据库迁移 | Alembic | 管理 MySQL 表结构版本 |
| 数据库 | MySQL 8 | 保存任务、数据集、模板、提交、审核、审计和导出数据 |
| MySQL Driver | PyMySQL | 与 SQLAlchemy 集成 |
| 后端包管理 | uv | 管理 API 和 Agent 的依赖与虚拟环境 |
| 前端包管理 | pnpm workspace | 管理 monorepo 前端依赖 |
| LLM 接入 | OpenAI API 兼容格式 | 通过环境变量配置 Key、Base URL 和模型名，不绑定供应商 SDK |
| 部署 | Docker Compose + Nginx | 一键启动 Web、API、Agent 和 MySQL |

## 3. 执行规则

- 前端页面、组件和状态管理不得引入与 Ant Design、`@dnd-kit/core`、Zustand 和自研模板运行时冲突的替代方案。
- 后端和 Agent 必须使用 Python，并通过 `uv` 管理包和虚拟环境。
- 数据库以 MySQL 为准，迁移脚本、索引和事务语义按 MySQL 运行环境设计。
- LLM 接入层必须按 OpenAI Chat Completions 兼容格式封装；真实密钥只能放在本地 `.env` 或部署环境变量中。
- 如供应商需要额外请求体参数，统一通过 `OPENAI_EXTRA_BODY_JSON` 或 `LLM_EXTRA_BODY_JSON` 显式配置，不在代码中写死供应商私有字段。
- 所有关键状态迁移由后端服务完成，前端只提交用户操作意图。
- 动态模板以 `TemplateSchemaVO` 为唯一运行时协议，Designer、Renderer、Labeler 作答和导出字段均复用该协议。

## 4. 与 SDD 的关系

- `apps/web/SDD.md` 约束前端页面、VO、接口调用和浏览器验收方式。
- `apps/api/SDD.md` 约束后端 Request、DTO、Entity、状态机、Agent 和导出契约。
- 前端 VO 字段名必须与后端返回 JSON 字段名一一对应。
- 后端对外部输入、模板 Schema、提交值和 LLM 输出均进行结构化校验。

## 5. 质量要求

| 维度 | 要求 |
| --- | --- |
| 类型安全 | 前端 TypeScript 与后端 Pydantic 共同约束接口字段 |
| 可追溯 | 任务、模板、提交、AI 预审、人工审核和导出均写入关键审计日志 |
| 可替换 | LLM Provider 可通过 OpenAI 兼容配置替换 |
| 可部署 | 本地开发和服务器部署均使用同一套 Compose 思路 |
| 可测试 | 单元、集成和真实浏览器验收分层覆盖 |
