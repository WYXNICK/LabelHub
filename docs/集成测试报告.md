# LabelHub 集成测试报告

生成日期：2026-06-09  
测试目标：依据《LabelHub 数据标注平台 AI 全栈课题实现要求》中完整链路要求，验证任务创建、数据导入、模板发布、标注作答、题目级 LLM 辅助、AI 预审、人工审核流转、数据验收和多格式导出等关键流程。

## 一、测试边界

集成测试用于验证多个模块之间的契约和状态流转，包括 API + 数据库会话、状态机、审计日志、文件生成、Agent worker 与模板运行时渲染。真实浏览器 E2E、真实 LLM 网络请求和真实 MySQL 方言兼容性不在本次自动化集成测试的完全覆盖范围内，详见“遗留风险”。

当前集成测试共 75 个用例：

| 模块 | 测试入口 | 用例数 | 说明 |
| --- | --- | ---: | --- |
| 前端 Web | `pnpm --filter @labelhub/web test:integration` | 6 | 模板运行时、Renderer、条件显示、提交值校验与隐藏字段裁剪 |
| 后端 API | `cd apps/api && uv run pytest tests/integration` | 67 | Owner、Labeler、Reviewer、AI 预审、导出全链路 |
| Agent | `cd apps/agent && uv run pytest tests/integration` | 2 | worker 成功写回与失败写回 |

## 五、测试中暴露的问题

本次最终测试未暴露业务代码缺陷。

测试重构过程中曾出现 1 类测试代码问题，已修正：

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| `test_dataset_import.py` 中 4 个导入用例找不到 `demo_data` 文件 | 测试文件从 `apps/api/tests/` 移动到 `apps/api/tests/integration/` 后，`ROOT_DIR = parents[3]` 指向了 `apps/`，导致读取 `apps/demo_data/...` | 改为 `parents[4]`，重新指向仓库根目录下的 `demo_data/` |

## 六、遗留风险与改进建议

| 风险 | 说明 | 建议 |
| --- | --- | --- |
| API 集成测试当前使用 SQLite 内存库 | 现有集成测试能验证业务状态机、API 契约和 ORM 逻辑，但不能完全证明 MySQL 方言、字符集、索引长度、事务隔离等生产数据库特性 | 后续增加一组 `tests/mysql_integration/` 或 pytest marker `mysql`，连接 Docker MySQL 的独立测试库运行关键 smoke：迁移、任务创建、导入、提交、AI 预审写回、人工审核、导出 |
| 前端集成测试不是浏览器 E2E | 当前前端集成测试以 React SSR/runtime 为主，未自动覆盖真实浏览器布局、网络请求和交互状态 | 对关键页面补充 Chrome DevTools MCP 或 Playwright 验收脚本：登录、Owner 任务/模板/导出、Labeler 作答、Reviewer 预审队列和人工审核 |
| Agent 集成测试使用 fake API/LLM | 可以验证 worker 调度和错误写回，但不会消耗真实 LLM，也不验证供应商网络稳定性 | 保留 fake 测试作为 CI 主路径；另设本地可选 smoke，通过环境变量显式开启真实 OpenAI 兼容 API 请求 |

## 七、结论

当前测试已经完成单元测试与集成测试的结构分离，并覆盖 LabelHub 当前实现的核心业务链路。自动化测试总计 153 个用例全部通过：

- 单元测试：78 个通过。
- 集成测试：75 个通过。
