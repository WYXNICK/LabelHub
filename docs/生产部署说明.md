# LabelHub 生产部署说明

本文档用于在一台已安装 Docker 的 Ubuntu 22.04 云服务器上部署 LabelHub。目标是：代码推送到 GitHub 后，在服务器拉取仓库，通过一条命令启动 Web、API、MySQL 和 Agent，并通过公网 IP 访问完整功能。

## 部署目标

- 服务器：Ubuntu 22.04 64 位，4 vCPU / 8 GiB，40 GiB 系统盘。
- 运行方式：Docker Compose 单机部署。
- 访问方式：公网 IP + 端口，不配置域名。
- LabelHub 默认公网端口：`18080`。
- 默认访问地址：`http://121.196.209.131:18080/`。

`18080` 避开了常见的 `80`、`443`、`8080` 等端口，适合在同一台服务器上与其他服务并存。如服务器已有端口规划，可在 `deploy/env/compose.env` 中修改。

## 服务结构

```text
公网访问 121.196.209.131:18080
        |
        v
labelhub-web  Nginx + React 静态资源
        |-- /api/* 反向代理到 labelhub-api:8000
        |-- /health 反向代理到 /api/health

labelhub-api  FastAPI + Alembic
        |
        v
labelhub-mysql MySQL 8.4

labelhub-agent 后台轮询 AI 预审 Job
        |
        v
OpenAI 兼容 LLM API
```

生产环境只暴露 Web 端口 `18080`。API、MySQL、Agent 均只在 Docker 内网访问，避免把数据库和后端端口暴露到公网。

## 部署文件

```text
deploy/
  compose.yaml              # 生产 Docker Compose
  docker/
    api.Dockerfile          # 后端 API 镜像
    agent.Dockerfile        # AI Agent 镜像
    web.Dockerfile          # 前端构建 + Nginx 镜像
  nginx/
    labelhub.conf           # 静态资源与 /api 反向代理
  env/
    .gitignore              # 忽略服务器本地密钥配置
    compose.env.example     # Compose 环境变量示例
    app.env.example         # 应用与 LLM 环境变量示例
  data/
    .gitignore              # 保留备份目录占位，不提交运行数据
scripts/
  deploy.sh                 # 一键部署脚本
```

## 首次部署

### 1. 放行安全组

在云服务器安全组入方向放行：

| 协议 | 端口 | 来源 | 说明 |
| --- | --- | --- | --- |
| TCP | 22 | 你的管理 IP 或 `0.0.0.0/0` | SSH 登录 |
| TCP | 18080 | `0.0.0.0/0` | LabelHub Web 访问 |

不需要放行 `3306`、`8000` 或 Agent 端口。

### 2. 拉取代码

```bash
cd /opt
git clone https://github.com/WYXNICK/LabelHub.git
cd LabelHub
```

如果仓库是私有仓库，请先在服务器配置 GitHub SSH Key 或 Personal Access Token。

### 3. 一键生成环境文件并启动

```bash
bash scripts/deploy.sh
```

脚本会自动创建：

- `deploy/env/compose.env`：公网端口、服务器公网 IP、MySQL 用户和密码。
- `deploy/env/app.env`：Session 密钥、Agent Token、LLM API 配置。

这两个文件包含密钥，已被 `.gitignore` 忽略，不能提交到 Git。

### 4. 配置 LLM API

首次运行脚本后，编辑 `deploy/env/app.env`：

```bash
nano deploy/env/app.env
```

至少填写为服务器环境中的实际配置，示例如下：

```bash
OPENAI_API_KEY=<server-runtime-secret>
BASE_URL=https://<openai-compatible-endpoint>/v1
MODEL_NAME=<model-name>
OPENAI_TIMEOUT_SECONDS=120
LLM_EXTRA_BODY_JSON=
```

保存后重启 API 和 Agent：

```bash
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml up -d --force-recreate api agent
```

说明：

- 后端和 Agent 都使用 OpenAI API 兼容格式，不绑定具体模型供应商。
- 不要把真实 `OPENAI_API_KEY` 写入仓库文件。
- 如果某个供应商需要额外 body 参数，可通过 `LLM_EXTRA_BODY_JSON` 配置 JSON 字符串。

### 5. 访问验证

```bash
curl -i http://121.196.209.131:18080/health
```

浏览器访问：

- Web 页面：`http://121.196.209.131:18080/`
- API 文档：`http://121.196.209.131:18080/api/docs`

默认 Demo 账号：

| 角色 | 邮箱 | 密码 |
| --- | --- | --- |
| Owner | `owner@labelhub.dev` | `labelhub123` |
| Labeler | `labeler@labelhub.dev` | `labelhub123` |
| Reviewer | `reviewer@labelhub.dev` | `labelhub123` |

## 日常更新

服务器上执行：

```bash
cd /opt/LabelHub
git pull
bash scripts/deploy.sh
```

`api` 服务启动时会自动执行：

```bash
uv run alembic upgrade head
```

因此数据库表结构会随代码迁移更新。

## 常用运维命令

以下命令均在仓库根目录执行：

```bash
# 查看服务状态
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml ps

# 查看全部日志
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs -f

# 只看后端日志
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs -f api

# 只看 Agent 日志
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs -f agent

# 重启某个服务
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml restart api

# 停止所有服务但保留数据卷
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml down
```

## 数据持久化与备份

生产部署使用 Docker volume 保存数据：

- `labelhub_mysql_data`：MySQL 数据。
- `labelhub_api_uploads`：导入文件、导出文件等 API 运行时文件。

建议在每次正式演示前做一次备份：

```bash
mkdir -p deploy/data/backups

# 备份 MySQL
docker exec labelhub-mysql sh -c 'mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" labelhub' \
  > deploy/data/backups/labelhub-$(date +%Y%m%d-%H%M%S).sql

# 备份上传和导出文件
docker run --rm \
  -v labelhub_api_uploads:/data \
  -v "$PWD/deploy/data/backups:/backup" \
  alpine tar czf /backup/labelhub-api-files-$(date +%Y%m%d-%H%M%S).tgz -C /data .
```

## 回滚

如果一次更新后出现问题：

```bash
cd /opt/LabelHub
git log --oneline -5
git checkout <上一个可用提交>
bash scripts/deploy.sh
```

确认恢复后，再决定是否切回主分支并修复问题：

```bash
git checkout main
git pull
```

## 排障

### 页面打不开

检查安全组是否放行 `18080`，再检查 Web 服务状态：

```bash
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml ps web
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs --tail=80 web
```

### 登录或接口提示服务不可用

检查 API 和 MySQL：

```bash
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml ps api mysql
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs --tail=120 api
```

常见原因：

- MySQL 尚未初始化完成。
- API 迁移失败。
- `deploy/env/app.env` 或 `deploy/env/compose.env` 被误删。

### AI 预审或题目级 LLM 辅助不可用

检查 Agent 和 LLM 配置：

```bash
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs --tail=120 agent
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs --tail=120 api
```

重点确认：

- `OPENAI_API_KEY` 不为空。
- `BASE_URL` 以 `/v1` 结尾。
- `MODEL_NAME` 与供应商实际模型名称一致。
- 服务器可以访问对应 LLM API。

### 端口冲突

如果 `18080` 已被占用，修改 `deploy/env/compose.env`：

```bash
LABELHUB_HTTP_PORT=18081
```

然后重新部署：

```bash
bash scripts/deploy.sh
```

同时在云服务器安全组放行新的端口。

### 重新部署时 `uv sync` 下载超时

如果执行 `bash scripts/deploy.sh` 时在 `RUN uv sync --frozen --no-dev` 阶段失败，并看到类似 `Failed to download distribution due to network timeout`、`Try increasing UV_HTTP_TIMEOUT` 的日志，通常是服务器到 Python 包源的下载速度较慢。

当前 API 和 Agent 镜像已在 Dockerfile 中设置：

```dockerfile
UV_HTTP_TIMEOUT=300
```

如服务器网络持续较慢，可再次执行部署命令，Docker 会复用已成功构建或下载的缓存：

```bash
bash scripts/deploy.sh
```

如果仍然失败，优先查看具体卡住的包名和耗时：

```bash
docker compose --env-file deploy/env/compose.env -f deploy/compose.yaml logs --tail=120 api
```
